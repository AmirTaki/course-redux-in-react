flux یک الکوی معماری (نه کتابخانه ) است که توسط فیس بوک معرفی شد تا مشکلات مدیرت State را در برنامه های بزرک حل میکند هدف اصلی Flux این است که جریان داده ها را در برنامه به صورت یک طرفه نکه دارد . این یعنی داده ها همیشه از یک مسیر مشخص حرکت میکنند

وظیفه ریداکس مدیریت state در پروژه است

اجزای اصلی معماری flux :

1- Action
2- Dispatcher
3- Store
4- View

Action :

چیزی است که در برنامه اتفاق می افتد, یک action شامل نوع رویداد (type) واطلاعات مربوط به آن (payload) است

Example : {type: 'ADD_TO_CART', payload: {productId: 1}}

ِDispatcher :

مانند یک پیام رسان است که action را به بخش های بعدی سیسیتم ارسال میکند

Sotre:

Store جایی است که State برنامه در آن ذخیره میشود

هرکاه یک Action از Dispatcher دریافت شود ، Store وضعیت را به روزرسانی میکند

مثلا اکر کاربر محصولی را به سبد خرید اضافه کند ،Store این محصول را به لیست خرید اضافه میکند

View:

view همان چیزی است که کاربر روی صفحه میبیند

view ها داده ها را از Store دریافت میکند و state به روز رسانی شده را نمایش میدهند

زمانی که Store تغییر کند ، View نیز به طور خودکار رندر میشود

جریان داده در flux (Flux Data Flow)

Flux از یک چرخه ساده و یک طرفه پیروی میکند:

1. کاربر کاری انجام می دهد (مثلا کلیک روی دکمه)

2. یک Action ساخته می شود و به Dispatcher ارسال می شود

3. Dispatcher این Action را به Store ارسال می کند

4. Store وضعیت را تغییر می دهد

5. View وضعیت جدید را از State دریافت کرده و رندر می شود.

این جریان یک طرفه باعث میشود که مدیریت وضیعت برنامه قابل پیش بینی تر و ساده تر است

ارتباط Flux با Redux

Redux ایده Flux را ساده تر و بهینه تر کرده است در واقع Redux نسخه بهبودیافته و مدرن تر Flux است

اگر چه Redux , Flux شباهت هایی دارند، Redux برخی از پیچیدگی های Flux را حذف کرده و استفاده از ان را راحت تر کرده است

شباهت های flux , redux

هر دو از جریان یک طرفه استفاده میکنند

هر دو از Action برای توصیف اتفاقات استفاده می کنند

State در هر دو معماری در مکانی مشخص نکهداری میشود

تفاوت های مهم Redux با Flux

فقط یک Stateدر Redux وجود دارد

در Flux می توان چندین Store داشت اما در Redux فقط یک Store وجود دارد که تمام state های برنامه را مدیریت می کند

Dispatcher در Redux حذف شده است

Redux نیازی به Dispatcher ندارد ، زیرا Actionها مستقیما به Store ارسال میشوند

تغییر state بکمک Reducer

در Store ، Flux مستقیما وضعیت را تغییر میدهند . اما در Redux ، تغییر وضعیت توسط Reducer انجام میشود

Immutable State

در State ، Redux به طور مستقیم تغییر نمی کند . به جای آن ، هر بار که تغییرات رخ می دهد، یک نسخه جدید از state ایجاد می شود . این کار تست پذیری را بسیار ساده تر می کند

معماری Redux

Actions -----> Reducer ------> Store -----> View -----> Action

State در Redux

همان طور که گفتیم ، store مسئول مدیریت state های برنامه است

ُState به داده هایی گفته می شود که نشاندهنده وضعیت برنامه هستند برای مثال داده های

سبد خرید (cart)

لیست محصولات (Shop)

وضعیت ورود کاربر (isLoggin)

چرا State مهم است

ُState نقش اصلی را در برنامه های تعاملی بازی میکند . هر تغییری که کاربر در برنامه ایجاد ممند مثل کلیک روی یک دکمه یا وارد کردن اطلاعات ، باید روی state اعمال شود . سپس برنامه با توجه به تغییرات state، رابط کاربری را به روز رسانی می کند

State در Redux

تمام دادهای برنامه در یک جا (store) نکهداری میشوند.

این کار به شما کمک میکند برنامه تان را پیش بینی پذیر کنید

ُState فقط قابل خواندن است : نمی توان مستقیما state را تغییر داد.

چرا تغییر مستقیم state مشکل ساز است ؟

عدم امکان ردگیری تغییرات :

در Redux، تمام تغییرات state باید از طریق reducer ها انجام شوند تا قابل پیش بینی و قابل درگیری باشند. با تغییر مستقیم نمیتوان مسیر تغییرات را به درستی دنبال کرد.

خرابی داده ها:

اگر چندین بخش از برنامه به داده یکسان دسترسی داشته باشند. تغییر مستقیم state ممکن است باعث بروز خطاها یا رفتار غیر منتظره شود.

چکونه State مدیریت میشود ؟

Redux از سه بخش اصلی برای مدیریت state استفاده میکند:

State:

جایی که داده های برنامه نگهداری میشوند.

Action:

دستورالعملی که می گوید چه تغییری باید در state اعمال شود

Reducer:

تابعی که تعیین می کند state چگونه تغییر کند

Action در Redux

یک Action توصیف میکند که چه تغییری باید روی state اعمال شود. در واقع ، Action نوعی پیام است که به store می گوید چه اتفاقی افتاده است.

هر Action شامل :

type (الزامی ): مشخص میکند که چه نوع تغییری قرار است رخ دهد

payload (اختیاری): داده های اضافی ای که برای انجام تغییرات لازم است.

Reudcer در Redux

Reudcer در Redux یک تابع خالص است که مسئولیت دارد تغییرات state را بر اساس اکشن های دریاقت شده مدیریت کند.

جریان داده در Redux

Redux به عنوان یک کتابخانه مدیریت حالت ، از یک اصل کلیدی تبعیت می کند، جریان داده ها همیشه یک طرفه است

جریان داده یکطرفه در Redux

در Redux ، داده ها از یک مسیر مشخص و یک طرفه عبور می کنندو هیج حلقه بازکشتی وجود ندارد.

این جریان به ترتیب زیر است:

1. شروع با یک Action

2. پردازش توسط Reducer

3. به روز رسانی Store

4. نمایش داده در UI

مفهوم Reducer

در Redux، یک reducer تابعی است که به عنوان ورودی state و action را دریافت می کندو یک state جدید بر میگرداند

مشکل Reducer بزرک

در پروژه های بزرگ ممکن است یک reducer به تنهایی برای مدیریت تمام state اپلیکیشن کافی نباشد.

چون :

مدیریت همهی بخش های مختلف اپلیکیشن مانند کاربران، محصولات، تنظیمات و غیره در یک فایل ممکن است باعث پیچیدگی زیاد شود.

مثال ساده

فرض کنید که ما یک اپلیکیشن داریم که باید دو بخش مختلف را مدیریت کند: کاربران و محصولات . به این صورت که:

یک بخش از state برای کاربران داریم.

یک بخش از state برای محصولات داریم.

در اینجا بجای یک reducer دو reducer ایجاد میکنیم و در نهایت از combineReducers استفاده می کنیم.

ترکیب چند reducer با combineReducers

در Redux از combineReducers استفاده می شود و این تابع به شما اجازه می دهد که چندین reducer را با هم ترکیب کنید، به طوری که هر reducer یک بخش خاص از state را مدیریت کند.
